# 文档审查规范技能

> 本技能定义Web3 SLG游戏项目中各类文档的审查标准、流程和质量要求。

## 技能概述

- **职责**: 文档审查 - 技能文档审查、项目文档审查、质量评估
- **适用场景**: 技术文档、需求文档、设计文档、API文档的审查与评估
- **技术栈**: React 19 + TypeScript 5.7 + Phaser 3.90 + Web3

## 文档体系架构

### 文档分类

项目文档分为四个层级：技能文档（Skills）、项目文档（Documents）、规则文档（Rules）、开发指南（AGENTS.md）。每个层级的文档有不同的受众和审查标准。

技能文档是AI助手的工作指南，定义特定角色的职责、流程和最佳实践。技能文档的主要受众是AI助手，需要清晰、具体、可操作的指导。项目文档记录项目的规划、设计、进度等信息，主要受众是项目团队成员。规则文档定义项目的技术规范和编码标准，是开发工作的参考依据。开发指南是项目的入口文档，整合所有重要信息。

不同层级文档的审查重点不同。技能文档重点审查职责定义是否清晰、流程是否完整、示例是否准确。项目文档重点审查内容完整性、逻辑一致性、时效性。规则文档重点审查规范合理性、与代码的一致性、可执行性。

### 文档关系

文档之间存在相互引用和依赖关系。AGENTS.md作为入口文档，引用技能文档和项目文档。技能文档引用规则文档中的编码标准和规范。项目文档引用技能文档作为开发依据。审查文档时需要考虑这些关系，确保引用正确、版本一致。

文档的版本管理应当与代码版本管理一致。重要文档的变更应当记录版本历史。文档与代码应当同步更新，避免文档与实现脱节。文档的废弃应当有明确的标识和替代说明。

## 技能文档审查

### 审查维度

技能文档的审查从六个维度进行：职责定义、流程完整性、规范准确性、示例有效性、可操作性和时效性。每个维度有具体的审查标准和评分规则。

职责定义维度审查技能文档是否清晰定义了AI助手的角色定位、工作范围、职责边界。职责定义应当具体、可衡量，避免模糊的描述。职责之间应当有清晰的边界，避免职责重叠或遗漏。

流程完整性维度审查技能文档定义的流程是否完整。流程应当包含开始条件、执行步骤、决策点、结束条件。关键流程应当有异常处理和回退机制。流程应当与项目实际工作方式一致。

### 审查标准

#### 职责定义审查标准

职责定义应当满足以下要求：角色定位清晰，说明技能所属的专业领域；工作范围明确，列出职责覆盖的具体工作内容；职责边界清晰，说明不包含的工作内容；成功标准可衡量，定义职责履行的评判标准。

以下是不合格的职责定义示例：

```
职责：负责代码开发
```

这个定义过于模糊，没有说明具体负责什么类型的代码开发。正确的定义应该是：

```
职责：负责游戏系统开发 - 包括英雄系统、战斗系统、资源系统的需求分析、功能设计、代码实现、测试配合
```

#### 流程完整性审查标准

流程定义应当满足以下要求：有明确的触发条件，说明什么情况下执行流程；有完整的步骤序列，每步骤有明确的操作内容；有决策点分支，说明不同情况下的处理方式；有明确的结束条件，说明流程何时完成；有异常处理机制，说明错误情况的处理方式。

流程示例：

```
## 英雄创建流程

### 触发条件
用户点击"创建英雄"按钮

### 执行步骤
1. 验证用户资源是否足够
2. 随机生成英雄属性
3. 分配英雄ID
4. 保存到数据库
5. 返回创建结果

### 决策点
- 资源不足：返回错误提示，流程结束
- 生成成功：继续执行步骤3

### 异常处理
- 数据库错误：回滚操作，返回系统错误提示
- 网络超时：重试3次，失败后返回超时提示
```

### 审查清单

技能文档审查使用以下检查清单：

| 审查项     | 审查内容                         | 权重 |
| ---------- | -------------------------------- | ---- |
| 角色定位   | 是否清晰说明专业领域和角色类型   | 15%  |
| 职责范围   | 是否明确列出具体工作内容和边界   | 20%  |
| 流程完整性 | 是否包含完整的工作流程和异常处理 | 25%  |
| 规范准确性 | 引用的规范是否与项目规范一致     | 15%  |
| 示例有效性 | 示例代码是否准确、可运行         | 15%  |
| 时效性     | 文档是否与当前项目状态一致       | 10%  |

## 项目文档审查

### 审查要点

项目文档审查关注四个方面：内容完整性、逻辑一致性、时效性和可读性。不同类型的项目文档审查重点不同。

需求文档重点审查需求描述的完整性、无歧义性、可实现性。每个需求应当有明确的业务价值、验收标准、优先级。需求之间应当一致，不存在逻辑矛盾。需求应当可实现，技术风险应当在可控范围内。

设计文档重点审查设计方案的合理性、可扩展性、可维护性。设计应当满足需求，有清晰的架构描述。设计方案应当考虑未来的扩展需求。设计应当遵循项目架构原则，与现有系统保持一致。

进度文档重点审查信息的准确性、时效性、可追溯性。进度数据应当真实反映项目状态。进度应当定期更新，与实际进展保持一致。重要变更应当记录原因和影响。

### 审查流程

项目文档审查流程包括：提交审查、初步筛选、详细审查、问题反馈、修订确认、审批通过六个环节。

提交审查时，文档作者应当说明文档类型、审查重点、紧急程度。初步筛选检查文档格式是否规范、是否完整、是否有明显的错误。详细审查按照审查要点进行全面检查。问题反馈将发现的问题汇总，返回给文档作者。修订确认验证作者是否正确修改了问题。审批通过后文档正式生效。

### 常见问题

项目文档常见问题包括：内容不完整，缺少必要的章节或信息；逻辑不一致，前后描述相互矛盾；时效性差，文档内容与实际状态不符；格式不规范，难以阅读和理解；可操作性差，缺少具体的执行指导。

审查发现问题时，应当区分问题类型：严重问题必须修改，如内容错误、遗漏重要信息；一般问题建议修改，如格式问题、表述不清；可选问题可以不改，如优化建议、个人偏好。

## 规则文档审查

### 审查原则

规则文档定义项目的技术规范和编码标准，是开发工作的参考依据。规则文档的审查重点是：规范合理性、与代码的一致性、可执行性、时效性。

规范合理性审查规则的必要性和合理性。规则应当解决实际问题，不是为了规则而规则。规则应当与行业最佳实践一致。规则应当考虑团队实际情况，可执行性强。

与代码的一致性审查规则文档与实际代码是否一致。代码应当遵循文档定义的规范。如果代码无法遵循规范，应当修改代码而非放宽规范。规范与代码的差异应当有明确的原因说明。

### 编码规范审查

编码规范是规则文档中最核心的部分。编码规范审查包括：命名规范、代码结构、注释要求、错误处理等方面。

命名规范审查变量、函数、类、文件的命名是否一致、清晰、有意义。规范应当定义命名约定（camelCase、PascalCase等）、命名前缀后缀规则、缩写使用规则。命名应当有描述性，避免无意义的名称如temp、data。

代码结构规范审查代码的组织方式。规范应当定义文件结构、目录组织、导入排序、模块划分原则。代码结构应当清晰，便于理解和维护。

注释规范审查注释的使用是否恰当。注释应当解释为什么，而不是解释是什么。避免过度注释，代码本身应当尽可能自解释。重要的算法、业务逻辑应当有注释说明。

## 审查流程

### 审查发起

文档审查可以由文档作者发起，也可以由审查者主动发起。作者发起审查时，应当提交完整的文档和审查请求。审查请求应当说明文档类型、审查重点、期望完成时间。

审查请求模板：

```
## 审查请求

### 文档信息
- 文档名称：
- 文档路径：
- 文档版本：
- 文档类型：技能文档 / 项目文档 / 规则文档

### 审查范围
- 主要审查点：
- 需要特别关注的问题：

### 时间要求
- 期望完成时间：
- 紧急程度：普通 / 紧急 / 加急
```

### 审查执行

审查者收到审查请求后，应当按照审查清单逐项检查。审查过程中应当记录发现的问题，区分问题类型和严重程度。审查完成后应当输出审查报告。

审查报告模板：

```
## 文档审查报告

### 基本信息
- 文档名称：
- 审查者：
- 审查日期：
- 审查结果：通过 / 需修改 / 需重写

### 审查摘要
- 严重问题数：
- 一般问题数：
- 建议数：

### 问题详情

#### 严重问题
1. [问题描述]
   - 位置：
   - 建议：
   - 优先级：P0 / P1

#### 一般问题
1. [问题描述]
   - 位置：
   - 建议：

#### 改进建议
1. [建议内容]

### 总体评价
[对文档的整体评价和改进方向]
```

### 问题跟踪

审查发现的问题应当跟踪处理。问题应当分配给具体的责任人。问题处理状态应当定期更新。问题关闭前应当验证修复是否正确。

问题状态包括：待处理、处理中、待验证、已关闭、已拒绝。每个状态有明确的流转规则。待处理问题应当尽快分配和处理。处理中的问题应当有进展更新。待验证问题应当及时验证。

## 质量评估

### 评分体系

文档质量使用百分制评分。评分维度包括：完整性、准确性、清晰性、一致性、时效性。每个维度满分20分，总分100分。

完整性维度评估文档内容是否完整。检查是否包含所有必要的章节，信息是否全面。完整的内容应当包含概述、主体内容、示例、参考等部分。

准确性维度评估文档内容是否正确。检查事实是否准确，技术描述是否正确，引用是否正确。错误的信息会误导读者，造成严重后果。

清晰性维度评估文档是否易于理解。检查表述是否清晰，结构是否合理，格式是否规范。清晰的文档能够让读者快速找到需要的信息。

一致性维度评估文档内部和文档之间是否一致。检查术语使用是否统一，格式是否一致，引用是否一致。不一致的文档会让读者困惑。

时效性维度评估文档是否与当前状态同步。检查文档是否过期，内容是否最新，版本是否正确。过时的文档可能导致错误的行为。

### 质量标准

文档质量分为四个等级：优秀（90-100分）、良好（75-89分）、合格（60-74分）、不合格（60分以下）。

优秀文档应当满足：内容完整准确，结构清晰合理，示例可运行有效，规范与实践一致，时效性良好。优秀文档可以作为项目的标杆文档。

良好文档应当满足：内容基本完整，存在一些小问题但不影响使用，结构较为清晰，存在少量不一致。良好文档可以投入使用，但应当逐步改进。

合格文档应当满足：包含主要内容，存在一些问题但不影响理解，需要改进但不是必须。合格文档需要作者根据反馈进行修改。

不合格文档应当满足：内容严重缺失或错误，无法正常使用，必须进行大幅修改。不合格文档需要重新编写。

## 审查工具

### 自动化检查

文档审查可以借助自动化工具提高效率。语法检查工具可以检查文档的语法错误。格式检查工具可以检查文档的格式规范。链接检查工具可以检查文档的内部链接和外部引用。

Markdown文档可以使用markdownlint进行格式检查。检查规则包括标题层级、列表格式、代码块格式、链接格式等。markdownlint可以集成到CI流程中，每次提交自动检查。

技术文档可以检查代码示例的正确性。代码示例应当保持可运行，与文档描述一致。可以使用脚本定期验证代码示例。

### 协作工具

文档审查需要团队协作完成。可以使用Git进行文档版本管理。文档的审查和修改通过Pull Request进行。审查意见通过代码评论功能记录。

协作平台应当提供：文档共享、版本历史、评论功能、任务跟踪。常见的文档协作平台包括GitHub、GitLab、Confluence、Notion等。

## 最佳实践

### 审查时机

文档审查应当在关键节点进行。新文档完成后应当进行审查。文档重大修订后应当进行审查。定期对存量文档进行复审。遇到重要决策前应当审查相关文档。

文档与代码应当同步审查。代码审查时应当检查相关文档是否需要更新。文档审查时应当检查文档描述与代码实现是否一致。

### 审查分工

文档审查应当有明确的分工。技能文档由具备相关技能的开发者审查。项目文档由项目负责人或指定人员审查。规则文档由技术负责人审查。重要文档应当由多人审查。

审查者应当具备：相关领域的知识、文档编写经验、审查技能。审查者应当客观公正，实事求是。审查者应当按时完成审查任务。

### 持续改进

文档审查发现的问题应当用于改进。常见问题应当汇总分析，识别根因。共性问题应当制定规范，从源头减少问题。审查标准应当根据实际情况调整优化。

文档质量应当作为团队绩效考核的参考。文档质量优秀的应当给予表彰。文档质量问题的应当进行改进指导。
