# `.trae/specs` 目录说明

本目录用于存放**任务执行过程中的结构化规格文档**，通常按一个主题或一次整改建立一个子目录。

## 典型结构

每个 spec 目录通常包含：

- `spec.md`：问题定义、目标、约束、方案
- `tasks.md`：执行拆解与状态跟踪
- `checklist.md`：验收项与回归检查项

## 目标读者

- 需要按任务推进实现的开发者
- 需要回溯某次整改过程的协作者
- 需要理解某轮迭代背景的 AI / 自动化代理

## 使用边界

- `specs` 反映的是某次任务过程，不天然等于当前长期标准
- 若任务完成后形成稳定做法，应同步沉淀到 `docs/` 或 `.trae/rules/`
- 若 `specs` 与当前代码不一致，应优先信任代码事实

## 当前目录现状

当前已存在的过程目录包括：

- `alliance-system-impl`
- `fix-build-errors`
- `fix-eventbus-missing`
- `implement-game-modes`
- `main-city-dev-plan`
- `project-optimization-audit`

为避免误读，建议在本目录内统一按以下状态理解：

- **已完成整改记录**：用于回溯一次已收口的问题修复，不代表当前仍存在同类问题
- **已完成实现过程文档**：用于回溯某轮功能实现拆解，不替代当前稳定能力说明
- **历史开发计划**：包含未完成条目、旧假设或旧目录结构，不等于当前开发路线
- **历史审计记录**：记录某次评估结论，部分勾选项或判断可能已随代码演进失效

## 维护建议

- 新建 spec 时保持 `spec / tasks / checklist` 三件套
- 任务结束后补充结果或结论，避免只有计划没有收口
- 若目录内容已完全过时，可在目录内补归档说明
