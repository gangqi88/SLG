# 规则与技能统一治理 Spec

## 背景

当前仓库已经具备「规则分片 + 调度入口」基础，但对于日常开发中的关键治理原则（最小变更、灰度发布、兼容优先、可逆上线、流程共识）尚未形成统一且可执行的闭环。结果是：

- 原则分散在经验描述中，缺少可直接执行的规则条目
- 核心技能文档对这些原则覆盖不均衡，协作接口不一致
- 文档审查与代码审查缺少统一验收口径

## 目标

将“谨慎保守、稳定优先”的工程原则沉淀为项目统一规范，并映射到关键技能，使其可被调度、可被执行、可被审查。

## 变更范围

### 规则层（`.trae/rules/project-rules/`）

- 新增“变更与发布安全”规则分片（建议：`R6-change-safety.md`）
- 更新规则索引 `SKILL.md`，纳入 R6 路由入口与优先级说明
- 必要时补充 `R3-version.md` / `R5-review.md` 的审查口径（仅与统一治理直接相关）

### 技能层（`.trae/skills/`）

- 更新 `task-scheduler/SKILL.md`：在路由阶段显式加入风险分级、灰度要求、回滚要求
- 更新 `fullstack-code-standards/SKILL.md`：落地原子化变更、兼容迁移、防御性编程、测试门禁
- 更新 `game-tester/SKILL.md`：补充灰度观测、契约测试、回滚/前向恢复验证项
- 更新 `document-reviewer/SKILL.md`：补充“原则一致性”审查维度与否决条件
- 视需要同步 `skills/SKILL.md` 索引中的治理说明（不重复正文）

## 不在本次范围

- 不改业务功能代码
- 不引入新的发布平台或监控系统实现
- 不重构历史技能体系，仅做统一治理对齐

## 统一治理原则映射

### 1) 变更粒度：原子化与正交性

- 单次提交/合并请求只做一类变更
- 大功能必须采用特性开关，实现“部署与发布解耦”
- 以增量演进替代大规模重写，优先隔离风险

### 2) 发布策略：灰度与渐进

- 发布默认采用金丝雀/渐进放量
- 监控与日志必须先于上线准备完成
- 业务逻辑变更优先使用 A/B 测试验证收益

### 3) 代码质量：防御与兼容

- API/数据结构变更遵循读写双向兼容
- 数据库迁移遵循 Add -> Backfill -> Deploy -> Use -> Drop
- 外部输入默认不可信，使用防御性编程兜底
- 核心逻辑维持高质量单元测试，跨服务维持契约测试

### 4) 风险预案：可逆性

- 任何改动必须具备快速回滚路径
- 对不可逆变更提供前向恢复方案与补偿操作

### 5) 团队协作：共识与流程

- 代码审查聚焦“最小化变更 + 影响评估”
- 中大型改动先产出技术设计文档并评审
- 禁止“经验直觉上线”，必须以监控与测试证据支撑

## 规范产出要求

- 新增规则分片采用中文（简体）并保持短句可执行表达
- 每份被修改技能文档需补齐版本尾注（版本号、最后更新、相关文档）
- 全局规则不在技能内重复粘贴，技能只描述自身执行动作
- 所有“必须”项在文档审查清单中有对应检查点

## 验收定义

1. 规则索引可路由到统一治理分片，且无冲突描述
2. 四个核心技能均新增统一治理执行段落（调度/开发/测试/审查）
3. 审查清单可直接用于判定是否满足“最小变更、灰度发布、兼容、可回滚”
4. 文档之间引用关系清晰，无重复维护同一规则正文

## 风险与应对

- 风险：规则过重导致执行成本上升  
  应对：明确“按变更级别触发”的分层要求，避免一刀切
- 风险：技能文档重复膨胀  
  应对：坚持“规则在 rules，动作在 skills”，交叉引用替代复制
- 风险：历史文档与新规则并存引起误读  
  应对：在受影响文档中增加“现行真源”与版本更新说明
