# 主城（Main City）系统开发计划规范

## 1. 项目概述

### 1.1 项目背景与目标

本开发计划基于 `MAIN_CITY_PLAN.md` 文档中的完整功能规格，设计并实现主城（世界中心）系统。主城作为全服玩家社交与交易的核心枢纽，需要实现建筑NFT化、预言机实时价格喂价、随机数公平性保障（VRF）、以及拍卖行智能合约自动化等功能。项目需严格对齐 `Web3游戏项目-SLG.md` 中第10.4节建筑系统规范，确保与整体SLG游戏架构保持一致。

项目成功的关键指标包括：建筑升级功能响应时间低于500毫秒、预言机价格更新延迟不超过5分钟、拍卖行交易实时更新、智能合约测试覆盖率不低于90%。主城系统是玩家日常游戏体验的核心场所，其稳定性和性能直接影响用户留存率。

### 1.2 文档关联性

本开发计划与以下核心文档保持对齐关系：首先是 `MAIN_CITY_PLAN.md`（版本2.0.0），该文档完整定义了建筑类型（城堡、仓库、城墙等14种建筑）、升级规则、数值参数、链上交互接口等技术规格；其次是 `Web3游戏项目-SLG.md`，该文档定义了整体SLG游戏的资源系统（木材、石料、粮食、铜币）、英雄养成体系、商业化系统以及Web3集成方案；最后是 `ALLIANCE_PLAN.md`，主城系统需要与联盟系统进行数据交互，包括联盟贡献点竞价广告位、联盟商店资源兑换等功能联动。

## 2. 项目里程碑

### 2.1 里程碑定义

项目共划分为五个主要里程碑，每个里程碑包含明确的可交付成果和验收标准。里程碑之间存在依赖关系，后续里程碑必须在前序里程碑通过验收后方可启动。

**里程碑M1：基础设施与核心架构（预计第1-2周）**

本里程碑的核心目标是建立主城系统的技术基础设施，包括TypeScript类型定义、状态管理架构、React组件基座、以及智能合约的基础接口定义。可交付成果包括：MainCity.ts类型文件（包含Building、AdBid、ShopItem、AuctionItem等接口定义）、MainCityManager单例类框架（包含所有API方法签名）、主城React组件目录结构搭建完成、智能合约接口文件（IBuildingNFT.sol、IAuctionHouse.sol、IPriceOracle.sol）。验收标准是所有TypeScript文件编译无错误，智能合约接口能够通过Solidity编译器0.8.19版本的语法检查。

**里程碑M2：建筑系统实现（预计第3-5周）**

本里程碑实现核心的建筑升级功能，包括前端的建筑卡片展示、升级弹窗交互、升级倒计时逻辑、以及后端的资源扣除和属性生效逻辑。可交付成果包括：MainCityView主城视图组件（展示所有建筑）、BuildingCard建筑卡片组件（支持点击升级）、BuildingUpgrade升级弹窗组件（展示消耗资源、确认升级）、MainCityManager建筑相关方法完整实现（getBuilding、startUpgrade、cancelUpgrade、completeUpgrade、getUpgradeCost）、建筑升级数值配置表（14种建筑每等级的木材、石料、铜币、时间消耗）、localStorage持久化逻辑。可交付成果的验收标准是：建筑升级响应时间低于500ms、升级倒计时精确到秒、资源扣除正确、升级完成后属性生效、建筑数据能够正确保存和加载。

**里程碑M3：广告牌与官方商店（预计第6-7周）**

本里程碑实现广告牌竞拍系统和官方商店购买系统。广告牌系统需要支持联盟贡献点竞价、周结算逻辑、广告展示等功能；官方商店需要支持商品展示、购买扣款、限购逻辑、折扣计算等功能。可交付成果包括：AdBillboard广告牌组件（展示当前广告和竞价入口）、OfficialShop官方商店组件（商品网格展示）、AdBiddingModal竞价弹窗组件、ShopPurchaseModal购买确认弹窗、MainCityManager广告和商店方法实现（placeAdBid、resolveWeeklyAdWinner、getShopItems、purchaseItem）、广告竞价状态机实现、商店商品数据配置。可交付成果的验收标准是：广告周结算正确触发并更新展示内容、商店购买成功后道具正确到账、限购逻辑正确执行、折扣计算准确。

**里程碑M4：拍卖行系统（预计第8-10周）**

本里程碑实现完整的拍卖行系统，包括拍卖品展示、出价竞拍、一口价购买、自动结算等功能。同时需要完成预言机喂价系统的链上合约开发。可交付成果包括：AuctionHouse拍卖行主组件（物品列表）、AuctionItem拍卖品卡片组件（展示当前价格和倒计时）、AuctionBidModal出价弹窗组件、AuctionBuyoutModal一口价弹窗组件、MainCityManager拍卖方法实现（getAuctionItems、placeBid、buyout、createAuction、cancelAuction）、PriceOracleAggregator.sol聚合价格预言机合约、RandomOracle.sol随机数预言机接口合约、预言机前端集成逻辑（定时拉取价格数据）。可交付成果的验收标准是：出价后价格正确更新、一口价立即成交、拍卖超时自动结算、预言机价格数据正确获取并显示。

**里程碑M5：Web3集成与全面测试（预计第11-12周）**

本里程碑完成所有Web3集成工作，包括建筑NFT铸造、钱包连接、交易签名等。同时进行全面的测试和Bug修复，确保系统稳定上线。可交付成果包括：BuildingNFT.sol完整智能合约实现（继承ERC721URIStorage）、AuctionHouse.sol完整拍卖行合约、MainCityWeb3集成层（钱包连接、交易签名）、Hardhat测试脚本（本地网络和Goerli测试网）、测试覆盖率报告（≥90%）、主城系统与联盟系统数据联调、线上部署配置和部署脚本。可交付成果的验收标准是：所有智能合约通过测试并部署成功、前端Web3功能正常运行、测试覆盖率≥90%、系统通过压力测试（100并发用户模拟）、所有已知Bug修复完成。

## 3. 技术选型与架构设计

### 3.1 前端技术栈

前端采用React 18 + TypeScript 5技术栈，这与项目现有的技术体系保持一致。状态管理使用Zustand（轻量级、TypeScript友好），相比Redux具有更简洁的API和更好的性能表现。UI组件库采用自定义组件+CSS Modules方式，确保与游戏整体视觉风格保持一致，同时避免第三方UI库带来的样式定制成本。

组件架构采用原子设计模式（Atomic Design），将组件划分为原子（Button、Input等基础组件）、分子（BuildingCard等组合组件）、有机体（MainCityView等复杂组件）、模板（MainCityPage等页面模板）、页面（实际的React Router页面）五个层级。这种划分方式有助于组件的复用和维护。组件通信使用自定义EventBus，对于跨层级的状态同步使用Zustand的store共享。

### 3.2 后端与存储架构

数据持久化采用localStorage作为主要存储方案，这是Web3游戏常见的前端主导架构模式。数据序列化使用JSON格式，对于频繁访问的数据（如当前资源数量）使用Zustand的persist中间件实现自动持久化。后续可扩展支持IndexedDB以存储更大的数据集。

智能合约开发使用Hardhat作为开发框架和本地测试网络，部署目标为Fractal Bitcoin（基于Bitcoin的Layer 2）。Solidity编译器版本使用0.8.19，该版本提供了良好的Gas优化和安全性保证。智能合约使用OpenZeppelin的ERC721URIStorage作为建筑NFT的基础实现，参考其经过审计的安全代码库。

### 3.3 链上交互架构

建筑NFT采用ERC721标准，每个建筑对应一个NFT Token，Token元数据存储在IPFS（通过Pinata服务），链上仅存储元数据URI。预言机系统采用多源聚合方案，从三个不同的价格数据源获取资源价格，取中位数作为最终价格，以防止单点操纵。随机数使用Chainlink VRF（Verifiable Random Function），确保链上随机数的公平性和可验证性。

前端与链上交互使用Ethers.js（v6）库，该库提供TypeScript类型支持，与钱包（MetaMask、UniSat等）的集成更加便捷。交易签名采用EIP-712标准，确保用户明确理解交易内容。

## 4. 任务分解与进度安排

### 4.1 第一阶段：基础设施（第1-2周）

第1周的任务包括：创建项目目录结构和TypeScript类型定义。具体工作项有：在src/types/目录下创建MainCity.ts，定义Building、BuildingType枚举、BuildingUpgradeCost、AdBid、AdSpace、ShopItem、AuctionItem等接口；在src/components/目录下创建MainCity/目录结构，包含MainCityView.tsx、BuildingCard.tsx、BuildingUpgrade.tsx等6个组件文件的框架代码；在src/game/logic/目录下创建MainCityManager.ts单例类框架，包含所有方法的空实现和TypeScript类型标注。第1周的交付成果是类型定义文件和组件框架文件，验收标准是TypeScript编译无错误。

第2周的任务包括：实现基础状态管理和localStorage持久化。具体工作项有：在MainCityManager中实现初始建筑数据生成逻辑（玩家初始拥有城堡、伐木场、采石场、农田、英雄府）；实现Zustand store用于状态管理；实现localStorage保存和加载方法；创建建筑升级数值配置文件（config/buildings.ts）。第2周的交付成果是可工作的基础状态管理逻辑，验收标准是页面刷新后建筑数据能够正确恢复。

### 4.2 第二阶段：建筑系统（第3-5周）

第3周的任务是实现建筑卡片和升级交互。具体工作项有：实现BuildingCard组件，显示建筑名称、等级、图标、升级按钮；实现BuildingUpgradeModal组件，显示升级消耗资源、当前资源、确认取消按钮；实现getUpgradeCost方法，根据建筑类型和当前等级计算升级消耗；实现startUpgrade方法，处理资源扣除和升级状态变更。第3周的交付成果是可交互的建筑卡片和升级弹窗，验收标准是点击升级按钮后资源正确扣除、升级状态正确显示。

第4周的任务是实现升级倒计时和完成逻辑。具体工作项有：实现升级倒计时显示逻辑（每秒更新）；实现completeUpgrade方法，升级时间到期后自动完成；实现cancelUpgrade方法，支持取消升级并返还50%资源；实现建筑属性生效逻辑（不同等级对应不同产出加成）。第4周的交付成果是完整的升级流程逻辑，验收标准是倒计时准确、资源返还正确、属性加成生效。

第5周的任务是优化用户体验和Bug修复。具体工作项有：实现升级加速功能（使用道具或付费加速）；实现批量升级队列；优化组件渲染性能（使用React.memo）；修复第3-4周发现的问题。第5周的交付成果是优化后的稳定版本，验收标准是响应时间低于500ms、无明显Bug。

### 4.3 第三阶段：广告牌与商店（第6-7周）

第6周的任务是实现广告牌系统。具体工作项有：实现AdBillboard组件，展示当前广告内容和联盟信息；实现AdBiddingModal组件，支持联盟出价；实现placeAdBid方法，处理出价逻辑和联盟贡献点扣除；实现resolveWeeklyAdWinner方法，每周结算选出最高出价者并退款给其他联盟。第6周的交付成果是可用的广告牌系统，验收标准是出价正确扣除贡献点、周结算正确执行。

第7周的任务是实现官方商店系统。具体工作项有：实现OfficialShop组件，商品网格展示；实现ShopItem组件，显示商品图片、名称、价格、限购信息；实现ShopPurchaseModal购买确认弹窗；实现purchaseItem方法，处理购买逻辑和货币扣除；实现限购逻辑，每周限购商品正确计算剩余数量。第7周的交付成果是可用的商店系统，验收标准是购买成功道具到账、限购正确执行。

### 4.4 第四阶段：拍卖行（第8-10周）

第8周的任务是实现拍卖行基础功能。具体工作项有：实现AuctionHouse组件，拍卖品列表展示；实现AuctionItem组件，显示当前价格、出价人数、剩余时间；实现getAuctionItems方法，获取活跃/已结束的拍卖品；实现placeBid方法，处理出价逻辑和价格更新。第8周的交付成果是基础拍卖功能，验收标准是出价后价格正确更新。

第9周的任务是实现一口价和结算功能。具体工作项有：实现buyout方法，一口价立即成交；实现createAuction方法，创建新的拍卖；实现cancelAuction方法，取消拍卖并返还物品；实现自动结算逻辑，拍卖超时后自动结束并转移物品给最高出价者。第9周的交付成果是完整的拍卖功能，验收标准是一口价立即成交、自动结算正确执行。

第10周的任务是实现预言机系统。具体工作项有：实现PriceOracleAggregator.sol聚合价格预言机合约；实现RandomOracle.sol随机数接口合约；实现前端价格展示组件，显示当前资源价格；实现定时拉取价格数据逻辑（每60秒）。第10周的交付成果是预言机集成，验收标准是价格数据正确显示和更新。

第11周的任务是实现智能合约和Web3集成。具体工作项有：实现BuildingNFT.sol完整合约（铸造、升级、转移、销毁）；实现AuctionHouse.sol完整合约；实现MainCityWeb3集成层（连接钱包、签名交易）；集成UniSat钱包（针对Fractal Bitcoin）。第11周的交付成果是可用的Web3功能，验收标准是钱包连接成功、交易签名正常。

## 6. 风险评估与应对措施

### 6.1 技术风险

**风险R1：Web3钱包集成复杂度超预期**

风险描述是不同钱包（MetaMask、UniSat）的API存在差异，可能导致集成工作量超出预期，影响整体进度。发生概率评估为中等（40%），影响程度为高。应对措施是首先集成最常用的MetaMask钱包作为主要目标，将UniSat钱包作为后续迭代功能；同时编写统一的钱桥接层（WalletBridge），屏蔽不同钱包的API差异。

**风险R2：预言机数据源可用性问题**

风险描述是外部价格数据源可能存在不可用或延迟的情况，影响资源价格展示的及时性。发生概率评估为中等（35%），影响程度为中。应对措施是实现多源聚合方案，当某一数据源不可用时自动切换到备用源；同时设置本地缓存，在数据源全部不可用时使用缓存数据；前端展示"价格更新中"状态，避免用户误解。

**风险R3：智能合约Gas优化不足**

风险描述是复杂的链上逻辑可能导致Gas消耗过高，增加用户交易成本，影响使用意愿。发生概率评估为低（20%），影响程度为高。应对措施是在开发阶段使用Hardhat的Gas Reporter插件监控每笔交易的Gas消耗；对高频操作（如建筑升级）使用批量处理减少链上交互次数；考虑使用Layer 2解决方案降低交易成本。

### 6.2 项目管理风险

**风险R4：需求变更频繁**

风险描述是业务方可能在开发过程中提出需求变更，影响进度和资源安排。发生概率评估为高（60%），影响程度为中。应对措施是在项目开始前与业务方明确需求范围，签署需求基线；建立正式的需求变更流程（Change Request），评估变更影响后再决定是否纳入迭代；保留一定的缓冲时间应对变更。

**风险R5：团队人员变动**

风险概率评估为低（15%），但影响程度为高。应对措施是确保代码文档完善，所有关键决策都有书面记录；实施结对编程，关键模块至少有两名成员熟悉；建立知识分享机制，每周技术分享会促进信息流动。

### 6.3 合规风险

**风险R6：虚拟商品合规性**

风险描述是各国对虚拟商品购买的法律规定不同，可能存在合规风险。发生概率评估为中（30%），影响程度为高。应对措施是在商店界面明确标注"虚拟商品，不支持退款"；针对不同地区设置不同的商品可用性；咨询法律顾问，制定符合主要目标市场（大陆、港澳台、东南亚）的合规方案。

性能测试需要达到以下指标：页面首次加载时间（LCP）低于2.5秒、建筑升级操作响应时间低于500毫秒、拍卖价格更新延迟低于1秒（WebSocket推送或轮询）、预言机价格数据更新延迟低于5分钟、前端内存占用峰值低于200MB。在模拟100并发用户的压力测试下，系统需要保持稳定运行，无内存泄漏或服务不可用情况。
