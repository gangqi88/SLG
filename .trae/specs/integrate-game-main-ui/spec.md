# 集成游戏主界面规格说明

> **当前归类**：进行中任务文档（待实现）  
> **使用边界**：本文用于定义“主界面集成”的目标与方案，不替代最终实现代码与验收结果。  
> **现行真源**：`src/router.tsx`、`src/components/Layout.tsx`、实际代码

## 背景与问题

当前应用没有面向玩家的“游戏主界面”。根路由 `/` 的 index 页面实际展示为英雄列表，顶部导航以开发期的模块入口为主，缺少：

- 主城视角的主界面承载（背景、状态栏、功能入口聚合）
- 统一的“回到主界面”落点（现有玩法退出会回到英雄列表）
- 关键系统入口（例如联盟系统）在路由层未集成

## 目标

- 提供一个“游戏主界面”作为默认落点，承担主城背景与主功能入口聚合。
- 将现有功能入口以“主界面菜单/按钮/九宫格”的形式集成到主界面。
- 统一退出落点：玩法页点击退出或触发退出事件后回到主界面。
- 保持现有路由仍可直接访问（便于调试与深链接）。

## 非目标

- 不重做各玩法内部界面与逻辑（例如战斗、采集、抽卡的交互细节不在本任务范围）。
- 不改动 Phaser 场景玩法规则（仅调整从主界面进入与退出返回的路径）。
- 不引入新的外部界面库或路由框架（沿用现有 React Router 与样式体系）。

## 现状梳理（事实）

- 路由入口：`src/router.tsx`
  - `/`（index）：HeroList
  - 已有功能页：`/city`、`/gathering`、`/tasks`、`/gacha`、`/lootbox`、`/tower-defense`、`/cooking`、`/siege`、`/battle` 等
- 页面壳层：`src/components/Layout.tsx`
  - 顶部 `NavLink` 导航更偏“开发模块入口”，并非游戏主菜单
- 已存在但未接入路由的主城 React 组件：`src/features/city/components/MainCityView.tsx`
- 多个玩法页退出行为统一回到 `/`（目前等价于回英雄列表）

## 方案概述

### 1）新增主界面页面（GameMain）

新增一个主界面页面（暂命名 `GameMain`），作为根路由 `/` 的 index 页面。主界面结构建议：

- 主城背景（优先复用 `public/game-assets/background/bg_city.svg` 或现有主城组件的视觉资产）
- 顶部状态栏（HUD）：钱包连接状态、基础信息（必要时预留资源条位）
- 功能入口区：以按钮/卡片/九宫格聚合进入各功能模块

功能入口至少包含：

- 英雄（原 HeroList）
- 主城/城建（现有 City / MainCity 相关）
- 资源采集（Gathering）
- 任务（Tasks）
- 抽卡/开箱（Gacha / LootBox）
- 战斗（Battle、守卫二乔、厨王争霸、攻城等）
- 联盟（现有 AllianceDashboard，补齐路由入口）

### 2）路由调整策略

- `/`（index）：指向新的 `GameMain`
- 将英雄列表从 index 迁移为显式路由（建议 `/heroes`），避免首页与功能页语义混淆
- 新增联盟路由（建议 `/alliance`），页面内容先接入 `AllianceDashboard`
- 现有路由保留不变，避免破坏已有入口与测试路径

### 3）统一“退出回主界面”

保持现有 “玩法页 onExit → navigate('/')” 行为不变，使退出落点从“英雄列表”自然变为“游戏主界面”。

### 4）Layout 从“开发导航”向“游戏壳层”迁移

`Layout` 的顶部 `NavLink` 导航建议调整为以下两种之一（实现时二选一）：

- **方案 A（推荐）**：移除顶部开发导航，改为游戏 HUD + 主界面内提供功能入口
- **方案 B（兼容）**：保留开发导航但迁移到独立路由（例如 `/dev` 或 `/style-guide`），主界面不展示开发导航

推荐优先方案 A，确保主界面观感更接近游戏产品态。

## 关键需求（可验收）

### 需求 1：主界面默认落点

系统应当在访问 `/` 时展示游戏主界面，而非英雄列表或开发导航页。

### 需求 2：功能入口集成

主界面应当提供可点击入口进入既有功能页，并在进入后不影响功能页原有行为。

### 需求 3：退出一致性

当玩家从 City/Gathering/LootBox/TowerDefense/Cooking/Siege 等玩法页退出时，应当回到主界面 `/`。

### 需求 4：联盟入口可用

主界面应当提供联盟入口，并确保联盟系统至少可通过路由访问与渲染（不要求链上能力完全可用）。

## 风险与约束

- 路由调整可能影响现有“首页=英雄列表”的预期，需要同步更新文档与端到端测试用例（如有）。
- 主界面视觉与布局优先复用现有资产与样式体系，避免新增依赖。

## 验证方式（建议）

- 基础：`npm run dev` 下人工验证主界面与各入口跳转
- 构建：`npm run build` 通过
- 回归：最少覆盖一次“从各玩法退出回主界面”的路径
