# 项目规则文档

> 本文档是所有技能文档的统一基础规范，定义语言要求、格式标准、版本管理和其他通用规则。所有技能文档必须遵守本文档的规定。

## 文档信息

- **文档编号**: PRJ-RULE-001
- **适用范围**: 所有技能文档（SKILL.md）
- **语言**: 中文（简体）

## 1. 语言规范

### 1.1 交流语言

- 所有技能文档使用**中文（简体）**编写
- 文档中的代码注释可以使用英文
- 技术术语首次出现时应当提供中文解释

### 1.2 标点符号

- 文档正文使用中文标点符号（，。：；？！""）
- 代码块中使用英文标点符号
- 列表项末尾不使用标点符号（除非是完整句子）

### 1.3 数字和单位

- 数字使用阿拉伯数字（1、2、3）
- 单位使用国际单位制或中文单位
- 示例：100ms、60fps、80级

### 1.4 术语统一

| 术语 | 使用 |
|------|------|
| 代码审查 | 不使用 code review |
| 持续集成 | 不使用 CI |
| 持续交付 | 不使用 CD |
| 单元测试 | 不使用 unit test |
| 接口 | 不使用 interface（TypeScript语境除外） |

## 2. 文档格式规范

### 2.1 文件命名

- 技能文档统一命名为 `SKILL.md`
- 目录使用 kebab-case：`web3-slg-project-manager`
- 技能文档路径：`.trae/skills/[skill-name]/SKILL.md`

### 2.2 文档结构

每个技能文档必须包含以下部分：

```markdown
# 技能名称

> 技能简述（一句话说明职责）

## 技能概述

- **职责**: 具体职责描述
- **适用场景**: 使用场景说明
- **技术栈**: 涉及的技术栈

## [主体内容]

### ...

## 相关文档

- 相关技能文档引用

---

*技能版本: X.Y.Z*
*最后更新: YYYY-MM-DD*
*相关文档: [引用列表]*
```

### 2.3 标题层级

- 一级标题：文档标题（#）
- 二级标题：主要章节（##）
- 三级标题：次要章节（###）
- 四级标题：细节说明（####）
- 不得超过四级标题

### 2.4 代码块

- 使用 ```typescript 或 ```bash 等语言标识
- 代码注释使用中文
- 保持代码简洁，避免过长

```typescript
// 正确示例
function calculateHeroPower(hero: Hero): number {
  const base = hero.attributes.command + hero.attributes.strength;
  return Math.floor(base * hero.starLevel * 0.1);
}
```

### 2.5 表格

- 表格标题使用中文
- 表头和内容使用中文
- 对齐清晰

| 列1 | 列2 | 列3 |
|------|------|------|
| 内容 | 内容 | 内容 |

### 2.6 列表

- 使用 `-` 或 `1.` 标记
- 同级列表保持一致的标记方式
- 列表项内容应当完整

## 3. 版本管理规范

### 3.1 版本号格式

采用语义化版本号：`主版本.次版本.修订号`

| 级别 | 何时递增 |
|------|----------|
| 主版本 | 重大结构调整、职责变更 |
| 次版本 | 内容重大更新、功能扩展 |
| 修订号 | 错误修正、格式调整、小幅更新 |

### 3.2 版本记录

每个技能文档末尾必须包含版本信息：

```markdown
---

*技能版本: 1.0.0*
*创建日期: 2026-01-01*
*最后更新: 2026-03-06*
*相关文档: doc1.md, doc2.md*
```

### 3.3 更新规则

- 文档内容变更必须更新版本号
- 修订号：格式调整、错别字修正、细微内容调整
- 次版本：新增章节、删除章节、内容重大修改
- 主版本：职责范围变更、文档结构重组

## 4. 技能文档通用要求

### 4.1 职责定义

每个技能文档必须明确定义：

- **核心职责**：技能的主要工作任务
- **适用范围**：在什么场景下使用该技能
- **边界定义**：不包含哪些工作

### 4.2 引用规范

技能文档之间应当相互引用，形成知识网络。

#### 引用格式

```markdown
相关文档：
- [文档名称] - 简要说明
- [自动化工作流] - CI/CD、构建部署
```

#### 引用规则

| 引用场景 | 引用方式 |
|----------|----------|
| 引用另一技能文档 | 使用文件名：fullstack-code-standards/SKILL.md |
| 引用外部资源 | 使用完整URL |
| 引用项目文档 | 使用相对路径：../documents/xxx.md |

### 4.3 职责边界

各技能的职责边界定义：

| 技能 | 核心职责 | 不包含 |
|------|----------|--------|
| 任务调度 | 任务分配、路由决策 | 具体执行 |
| 项目管理 | 规划、跟踪、决策 | 代码编写 |
| 代码规范 | 编码标准、技术方案 | 测试设计 |
| 测试规范 | 测试设计、质量保障 | 架构设计 |
| 自动化工作流 | CI/CD、工具开发 | 业务代码 |
| 文档审查 | 审查、质量评估 | 文档编写（部分除外） |

### 4.4 协作接口

技能文档应当定义清晰的协作接口：

- 输入：任务描述、参数
- 处理：执行流程
- 输出：结果格式、后续任务

```typescript
interface TaskInput {
  type: string;
  description: string;
  priority: 'P0' | 'P1' | 'P2' | 'P3';
}

interface TaskOutput {
  success: boolean;
  result: any;
  nextSkill?: string;
}
```

## 5. 内容规范

### 5.1 示例要求

- 代码示例必须可运行（或接近可运行）
- 示例应当使用项目实际的技术栈
- 示例注释使用中文

### 5.2 链接规范

- 内部链接使用相对路径
- 外部链接使用完整URL
- 链接文本应当描述性

### 5.3 图片规范

- 图片使用中文命名
- 图片存放在文档同目录的 `images/` 文件夹
- 使用Markdown图片语法

## 6. 审查规则

### 6.1 审查要点

| 审查项 | 要求 |
|--------|------|
| 语言 | 全部使用中文 |
| 格式 | 符合文档结构要求 |
| 版本 | 版本号正确更新 |
| 引用 | 相关文档正确引用 |
| 边界 | 职责边界清晰 |

### 6.2 审查流程

1. 作者完成文档编写
2. 检查是否符合本文档要求
3. 提交文档审查
4. 审查通过后正式发布

## 7. 常见问题处理

### 7.1 多技能交叉任务

当任务涉及多个技能领域时：

1. 由任务调度器识别任务类型
2. 确定主要负责技能
3. 明确协作顺序
4. 各技能按职责执行

### 7.2 文档冲突处理

当技能文档之间存在冲突时：

1. 以本文档（项目规则）为基准
2. 由文档审查技能仲裁
3. 更新相关文档解决冲突

### 7.3 新技能创建

创建新技能文档时：

1. 确认职责范围不与现有技能重叠
2. 遵循本文档的格式要求
3. 在AGENTS.md中添加引用
4. 更新任务调度器的路由规则
